# Rustitler MVP 需求文档

版本：0.1  
日期：2026-06-22  
状态：已确认需求草案  

## 1. 项目定位

Rustitler 是一个完全离线运行的桌面批量文件重命名工具。它面向中文办公文档场景，自动从 Word、PDF 和扫描件中识别文档主标题，并将文件复制到输出目录后按标题重命名。

MVP 第一版重点解决以下问题：

- 批量拖入文档后，自动提取真正的文档主标题。
- 高置信度文件自动复制输出并重命名。
- 低置信度、重复处理、提取失败和不支持格式进入待处理列表。
- 原文件始终不被修改。
- 全流程离线，不调用云端 API，不部署本地大模型。

## 2. MVP 目标

### 2.1 产品目标

- 提供 macOS 和 Windows 桌面应用。
- 支持批量拖入文件和文件夹。
- 支持 `.docx`、`.doc`、`.pdf`、`.png`、`.jpg`、`.jpeg`。
- 自动识别中文办公文档主标题。
- 使用主标题作为输出文件名。
- 自动处理重名文件，绝不覆盖已有文件。
- 提供待处理列表，允许用户手动编辑建议文件名并立即输出。
- 保存完整历史日志，并支持撤销本批次输出。
- 支持设置持久化、导入和导出。

### 2.2 技术目标

- 使用 Rust 开发核心后端逻辑。
- 使用 Tauri 构建桌面外壳。
- 使用 React + TypeScript 构建前端界面。
- PDF 坐标文本提取明确使用 `liteparse`。
- Word 纯文本提取明确使用 `undoc`。
- OCR 明确使用内置 Tesseract 和简体中文语言包。
- `.doc` 老 Word 格式通过内置转换组件转为可提取文本的中间格式。

### 2.3 离线约束

Rustitler MVP 的所有核心能力必须离线完成：

- 文件导入和扫描。
- PDF 文本和坐标提取。
- Word 文本提取。
- `.doc` 转换。
- 图片和扫描 PDF 的 OCR。
- 标题候选评分。
- 文件复制和重命名。
- 历史日志和设置存储。

应用不得调用云端 API，不得上传用户文件，不得依赖联网服务，不得部署或调用本地大模型。

## 3. MVP 非目标

以下能力不包含在 MVP 第一版中：

- 不做全文语义理解。
- 不部署本地大模型。
- 不接入任何云端 OCR、云端 NLP 或云端文件处理 API。
- 不做多行标题合并。
- 不渲染文档页面。
- 不高亮标题所在页面区域。
- 不递归扫描子目录。
- 不生成输出目录报告文件。
- 不处理 `.docm`、`.xls`、`.xlsx`、`.ppt`、`.pptx` 等其他格式。
- 不承诺 `.doc` 样式、版式和字体信息保真。
- 不做插件系统、模板市场或商业化功能。
- 不支持用户框选文档区域重新识别。

## 4. 关键概念

### 4.1 主标题

主标题指文档正文的主要标题，通常位于第一页或前几页上方，是最适合用作文件名的标题。例如：

- `关于召开年度工作会议的通知`
- `项目实施方案`
- `劳动合同`
- `管理制度`
- `情况报告`

MVP 输出文件名只使用主标题，不自动追加日期、编号、原文件名或来源单位。

### 4.2 候选标题

候选标题是解析阶段从文档中提取出的可能标题。候选标题会进入评分流程，并产生 0-100 的置信度。

### 4.3 置信度

置信度为 0-100 分。MVP 默认自动输出阈值为 70 分。

- 置信度达到 70 分及以上：自动复制输出并重命名。
- 置信度低于 70 分：进入待处理列表。

### 4.4 待处理列表

待处理列表用于承载无法安全自动输出的文件，包括：

- 标题低置信度。
- 标题提取失败。
- OCR 失败。
- `.doc` 转换失败。
- 文件格式不支持。
- 疑似重复处理。
- 文件读写或权限异常。

用户可以在待处理列表中直接编辑建议文件名，确认后立即复制输出该单个文件。

### 4.5 批次

一次拖入操作形成一个处理批次。批次日志用于历史查看和撤销输出。

## 5. 用户输入范围

### 5.1 文件拖入

用户可以拖入一个或多个文件。支持格式：

- Word：`.docx`、`.doc`
- PDF：`.pdf`
- 图片扫描件：`.png`、`.jpg`、`.jpeg`

不支持格式应加入队列并标记为跳过，同时显示原因。

### 5.2 文件夹拖入

用户可以拖入文件夹。MVP 只扫描文件夹第一层，不递归处理深层目录。

文件夹扫描规则：

- 只识别第一层内的支持格式文件。
- 第一层内不支持格式加入队列并标记为跳过。
- 子文件夹本身不递归扫描。
- 队列中必须显示来源路径，便于用户判断文件来自哪个目录。

## 6. 总体处理流程

```text
拖入文件或文件夹
  -> 扫描输入
  -> 过滤格式并加入队列
  -> 检查是否疑似重复处理
  -> 提取文本、坐标或 OCR 结果
  -> 生成候选标题
  -> 计算 0-100 置信度
  -> 高置信度自动复制输出并重命名
  -> 低置信度或异常进入待处理列表
  -> 写入应用历史日志
```

处理是实时自动的：用户拖入后立即开始分析和输出。低置信度和失败文件不会自动输出。

## 7. 技术架构

### 7.1 技术栈

- 桌面外壳：Tauri
- 后端：Rust
- 前端：React + TypeScript
- 构建工具：Vite
- PDF 提取：`liteparse`
- Word 提取：`undoc`
- OCR：内置 Tesseract + 简体中文语言包
- `.doc` 转换：内置离线转换组件

### 7.2 项目结构

MVP 从空目录启动，采用单个 Tauri 工作区，不在第一版拆分多 crate。

建议目录结构：

```text
Rustitler/
  doc/
    proposal.md
  src/
    components/
    pages/
    stores/
    types/
  src-tauri/
    src/
      commands/
      ingest/
      extract/
      scoring/
      rename/
      history/
      settings/
      errors.rs
      main.rs
    tauri.conf.json
  package.json
  vite.config.ts
  Cargo.toml
```

### 7.3 Rust 后端模块职责

#### `commands`

负责 Tauri 命令桥接，向前端提供导入、分析、输出、撤销、读取历史、保存设置等命令。

#### `ingest`

负责输入扫描和队列初始化：

- 处理拖入文件。
- 处理拖入文件夹。
- 只扫描文件夹第一层。
- 识别支持格式。
- 标记不支持格式。
- 计算重复处理检测所需的源路径、文件大小和修改时间。

#### `extract`

负责文档内容提取：

- PDF 通过 `liteparse` 提取带坐标文本。
- Word 通过 `undoc` 提取纯文本。
- `.doc` 通过内置转换组件转为可提取文本的中间格式。
- 图片通过 Tesseract OCR 提取文本块。
- 扫描 PDF 在无法提取原生文本时进入 OCR 兜底。

#### `scoring`

负责候选标题生成和评分：

- PDF 和图片使用坐标/几何评分。
- Word 使用纯文本启发式评分。
- 合并关键词、正则、文本质量和排除规则。
- 输出候选列表、分类分数、规则明细和最终置信度。

#### `rename`

负责输出副本创建和文件名处理：

- 生成输出目录。
- 复制原文件。
- 清洗文件名。
- 自动追加序号处理冲突。
- 确保不覆盖已有文件。

#### `history`

负责批次日志：

- 记录每个文件的源路径、输出路径、置信度、候选详情、状态和失败原因。
- 永久保留历史记录。
- 支持按批次撤销输出。

#### `settings`

负责用户设置：

- 读取和写入 `settings.json`。
- 保存自动输出阈值。
- 保存评分敏感度。
- 保存关键词和正则规则。
- 支持导入、导出和恢复默认设置。

## 8. 标题提取策略

MVP 使用务实双管线：

- PDF 和图片扫描件：坐标/几何识别优先。
- Word/DOCX/DOC：纯文本启发式识别。

### 8.1 PDF 标题提取

PDF 使用 `liteparse` 提取带坐标文本。

默认搜索范围：

- 先分析第一页。
- 如果第一页没有达到阈值的候选，再扩展到前 3 页。
- 不分析整篇文档。

候选单位：

- MVP 以单行为候选标题单位。
- 不做多行标题合并。

主要评分因素：

- 字号或视觉尺寸是否明显高于正文。
- 是否加粗。
- 是否位于页面上部。
- 是否水平居中或接近中轴。
- 是否处于合理标题区域。
- 是否命中中文办公关键词。
- 文本长度和标点是否像标题。
- 是否被识别为页眉、页脚、页码、编号、日期、密级或落款。

### 8.2 图片扫描件标题提取

图片扫描件支持：

- `.png`
- `.jpg`
- `.jpeg`

OCR 使用内置 Tesseract 和简体中文语言包。

识别策略：

- 对整张图片 OCR。
- 根据 OCR 文本块位置寻找页首标题。
- OCR 结果整体更保守。
- OCR 低置信度候选进入待处理列表。

### 8.3 扫描 PDF 标题提取

PDF 如果无法提取原生文本，或提取文本为空，应进入 OCR 兜底流程。

OCR 兜底要求：

- 不调用云端服务。
- 不调用本地大模型。
- OCR 失败时进入待处理列表。
- OCR 得到的候选标题不与原生文本同等置信，评分应更保守。

### 8.4 Word/DOCX/DOC 标题提取

Word 文件采用纯文本启发式，不在 MVP 中读取字号、居中、加粗等版式信息。

处理方式：

- `.docx` 使用 `undoc` 提取纯文本。
- `.doc` 先使用内置转换组件转为可提取文本的中间格式，再提取纯文本。
- 转换和提取只用于识别标题，输出仍保留原始格式。

候选范围：

- 默认从前 10 个非空段落中找候选。

评分因素：

- 段落是否靠前。
- 文本长度是否合理。
- 是否像正文长句。
- 是否包含过多标点。
- 是否是编号、日期、页眉、页脚或落款。
- 是否轻度命中中文办公关键词。

标题长度偏好：

- 默认偏好 2-40 个中文字符左右的短标题。
- 超长段落降分。

Word 由于缺少版式信息，置信度应天然更保守。

## 9. 打分与规则系统

### 9.1 默认评分模型

MVP 使用 0-100 分置信度模型，默认自动输出阈值为 70 分。

默认评分方向为“版式强优先”：

- PDF 和图片：版式和位置为主要判断依据。
- 关键词只作为辅助因素。
- 文本质量用于排除噪声和异常候选。
- OCR 候选整体更保守。
- Word 纯文本候选整体更保守。

文档不固定每条规则的精确分值，以避免早期过度承诺。实现时应保证分类权重方向符合上述原则。

### 9.2 分类分数

右侧候选详情需要展示分类分数：

- 版式分数。
- 位置分数。
- 关键词分数。
- 文本质量分数。
- 惩罚分数。

默认显示分类分数，展开后显示逐条规则明细。

### 9.3 高级敏感度设置

MVP 不暴露逐项规则分值。设置页提供少量分类敏感度滑块：

- 更重视版式。
- 更重视位置。
- 更重视关键词。
- 更重视文本质量。
- OCR 更保守。

滑块调整的是策略敏感度，不要求用户理解底层精确分值。

### 9.4 关键词和正则规则

MVP 支持用户在高级设置中添加关键词和正则表达式规则。

规则边界：

- 关键词和正则只做加减分。
- 不强制覆盖几何结果。
- 不提供“命中即标题”的强制规则。
- 默认不用配置也能运行。

默认中文办公关键词包括但不限于：

- `关于`
- `通知`
- `报告`
- `方案`
- `制度`
- `合同`
- `函`

### 9.5 排除规则

MVP 默认优先排除以下干扰内容：

- 页眉。
- 页脚。
- 页码。
- 密级。
- 编号。
- 日期。
- 落款单位。

这些排除规则可以体现为扣分或候选过滤，但不得导致标题明显缺失时应用崩溃。

## 10. 输出与重命名策略

### 10.1 输出方式

MVP 不原地重命名。所有高置信度文件都复制到输出目录后重命名。

原文件要求：

- 不修改。
- 不移动。
- 不删除。
- 不改变原文件格式。

### 10.2 输出目录

每个源目录旁创建一个固定输出目录：

```text
Rustitler 输出
```

如果一次拖入的文件来自多个不同目录，则每个源目录各自创建自己的 `Rustitler 输出`。

不保留源目录的相对层级。输出目录中只放该源目录对应处理出的文件。

### 10.3 输出文件格式

输出文件保留原始格式：

- `.doc` 输出仍为 `.doc`
- `.docx` 输出仍为 `.docx`
- `.pdf` 输出仍为 `.pdf`
- `.png` 输出仍为 `.png`
- `.jpg` 输出仍为 `.jpg`
- `.jpeg` 输出仍为 `.jpeg`

`.doc` 转换、PDF 提取和 OCR 只用于识别标题，不改变最终输出格式。

### 10.4 文件名清洗

文件名清洗策略为尽量保留原文。

规则：

- 替换 Windows 和 macOS 绝对非法字符。
- 保留中文标点。
- 保留书名号。
- 保留普通空格。
- 压缩明显异常的连续空白。
- 避免生成空文件名。
- 避免生成系统保留名称。

### 10.5 重名处理

如果输出目录中已经存在同名文件，MVP 自动追加序号，绝不覆盖。

示例：

```text
关于召开年度工作会议的通知.pdf
关于召开年度工作会议的通知 (2).pdf
关于召开年度工作会议的通知 (3).pdf
```

## 11. 重复处理检测

MVP 需要防止用户重复处理同一个源文件。

检测依据：

- 源文件路径。
- 源文件大小。
- 源文件修改时间。

如果再次拖入疑似已处理过的源文件：

- 自动跳过。
- 进入待处理列表。
- 显示“可能已处理过”的原因。
- 用户确认后才允许重新输出。

## 12. 日志与撤销

### 12.1 历史日志

完整历史日志保存在应用数据目录内，永久保留，由用户手动清理。

日志至少记录：

- 批次 ID。
- 批次创建时间。
- 源文件路径。
- 源文件大小。
- 源文件修改时间。
- 文件类型。
- 提取方式。
- 候选标题列表。
- 最终标题。
- 置信度。
- 分类分数。
- 规则明细。
- 输出路径。
- 执行状态。
- 失败原因。

输出目录不生成额外报告文件。

### 12.2 撤销定义

撤销指撤销本批次输出，即删除该批次生成的重命名副本。

撤销规则：

- 原文件始终不受影响。
- 如果输出副本仍存在，则删除该副本。
- 如果输出副本不存在，则跳过该项并提示。
- 如果输出副本已被用户改动，界面应提示风险并默认跳过该项。

## 13. 设置系统

### 13.1 设置存储

用户设置保存在应用数据目录中的配置文件，例如：

```text
settings.json
```

设置必须完全离线，可备份和迁移。

### 13.2 设置项

MVP 设置页至少包含：

- 自动输出阈值，默认 70。
- 版式敏感度。
- 位置敏感度。
- 关键词敏感度。
- 文本质量敏感度。
- OCR 保守度。
- 高级关键词规则。
- 高级正则规则。
- 设置导入。
- 设置导出。
- 恢复默认。

输出策略在 MVP 中固定为：

- 源目录旁创建 `Rustitler 输出`。
- 原文件不动。
- 复制副本并重命名。
- 输出保留原格式。

## 14. 界面需求

### 14.1 主界面布局

MVP 使用双栏界面。

左侧为文件队列，右侧为候选详情预览。

### 14.2 文件队列

文件队列需要展示：

- 文件名。
- 来源路径。
- 文件类型。
- 当前状态。
- 识别标题。
- 置信度。
- 输出结果。
- 失败原因。

状态至少包括：

- 分析中。
- 已输出。
- 待处理。
- 已跳过。
- 失败。
- 可撤销。

### 14.3 候选详情预览

右侧候选详情不渲染文档页面。

需要展示：

- 当前选中文件路径。
- 候选标题列表。
- 最终识别标题。
- 0-100 置信度。
- 分类分数。
- 展开后的逐条规则明细。
- 输出路径。
- 失败原因。
- 处理日志摘要。

### 14.4 待处理交互

待处理项允许用户直接编辑建议文件名。

确认后：

- 立即复制输出该单个文件。
- 自动处理重名序号。
- 写入历史日志。
- 更新队列状态。

### 14.5 历史页

历史页需要支持：

- 查看批次列表。
- 查看批次内每个文件的处理结果。
- 查看失败原因。
- 对已输出批次执行撤销。

## 15. 异常处理

MVP 必须对以下异常给出明确状态和原因：

- 不支持的文件格式。
- 文件读取失败。
- 文件权限不足。
- PDF 文本提取失败。
- PDF 无原生文本且 OCR 失败。
- OCR 引擎异常。
- `.doc` 转换失败。
- Word 文本提取失败。
- 没有找到可信标题。
- 置信度低于阈值。
- 疑似重复处理。
- 输出目录创建失败。
- 文件复制失败。
- 文件名清洗后为空。
- 撤销时输出副本不存在。
- 撤销时输出副本已被用户改动。

异常文件不得导致整个批次崩溃。队列中每个文件应独立记录状态。

## 16. 验收标准

### 16.1 样本文档集

MVP 使用 50 份中文办公样本文档验收：

- PDF：20 份。
- Word：20 份。
- 图片扫描件：10 份。

### 16.2 准确率目标

在 50 份样本文档中，高置信度自动命名准确率目标为：

```text
>= 90%
```

准确率只统计被系统判定为高置信度并自动输出的文件。低置信度进入待处理不计为误命名。

### 16.3 功能验收

MVP 必须满足：

- macOS 可运行。
- Windows 可运行。
- 可拖入多个文件。
- 可拖入文件夹并扫描第一层。
- 支持 `.docx`、`.doc`、`.pdf`、`.png`、`.jpg`、`.jpeg`。
- 不支持格式显示在队列中并标记为跳过。
- PDF 使用 `liteparse` 提取带坐标文本。
- Word 使用 `undoc` 提取纯文本。
- `.doc` 使用内置转换组件转为可提取文本的中间格式。
- 图片扫描件使用内置 Tesseract 简体中文 OCR。
- 高置信度文件自动复制输出并重命名。
- 低置信度文件进入待处理列表。
- 重名输出自动追加序号。
- 原文件不被修改。
- 历史日志永久保留。
- 可撤销本批次输出。
- 设置可保存。
- 设置可导入和导出。
- 全流程不依赖网络。

### 16.4 性能和可用性验收

MVP 不承诺具体处理秒数，但必须满足：

- 批处理过程中有进度状态。
- 批处理过程不能卡死界面。
- 用户可以取消正在进行的批处理。
- 单个文件失败不影响其他文件继续处理。

## 17. 依赖与打包风险

### 17.1 Tesseract

MVP 要内置 Tesseract 和简体中文语言包，确保开箱可用。

需要重点验证：

- macOS 打包可用性。
- Windows 打包可用性。
- OCR 语言包路径加载。
- 安装包体积。
- 许可合规。

### 17.2 `.doc` 转换组件

MVP 要内置 `.doc` 转换能力，目标是把 `.doc` 转成可提取文本的中间格式。

需要重点验证：

- 转换组件许可。
- 安装包体积。
- 静默调用稳定性。
- macOS 和 Windows 路径处理。
- 转换失败时的错误提示。

### 17.3 `liteparse`

PDF 提取必须验证：

- 是否能稳定提取文本内容。
- 是否能提供坐标或足够的版式信息。
- 是否能支持中文 PDF。
- 是否能识别页码和页面尺寸。
- 提取失败时是否能安全进入 OCR 兜底或待处理。

### 17.4 `undoc`

Word 提取必须验证：

- `.docx` 文本提取稳定性。
- 段落边界保留情况。
- 中文文本保留情况。
- `.doc` 中间格式输入的兼容性。
- 提取失败时的错误提示。

## 18. 后续可扩展方向

以下方向不属于 MVP，但架构应避免阻塞：

- 多行标题合并。
- 文档页面渲染和标题区域高亮。
- 自定义模板。
- 强制命中规则。
- 递归扫描子目录。
- 输出目录报告。
- 多语言 OCR。
- DOCX 样式解析。
- Word 转 PDF 后统一几何识别。
- 插件系统。
- 团队规则共享。
